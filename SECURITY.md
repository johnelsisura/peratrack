# 🔐 Security Policy

## Supported Versions

We actively maintain security updates for the following versions of **PeraTrack**:

| Version | Supported          |
| ------- | ------------------ |
| 5.1.x   | :white_check_mark: |
| 5.0.x   | :x:                |
| 4.0.x   | :white_check_mark: |
| < 4.0   | :x:                |

Only supported versions receive security patches and critical updates. Users are strongly encouraged to upgrade to the latest supported version.

---

## Reporting a Vulnerability

We take security issues seriously and value responsible disclosure from the community.

If you discover a security vulnerability in PeraTrack, please report it privately using the details below:

### 📩 Reporting Channel
Email: **security@peratrack.app**

If email is not available, you may also report via:
- GitHub Issues (ONLY if the issue is non-sensitive)
- Direct message to the repository maintainer (if applicable)

---

### 📌 What to include in your report
To help us validate and resolve the issue quickly, please include:

- Clear description of the vulnerability  
- Steps to reproduce the issue  
- Affected version(s)  
- Screenshots, logs, or proof of concept (if available)  
- Possible impact assessment (optional but helpful)

---

### ⏱ Response Timeline
We aim to respond within the following timeframe:

- **Acknowledgement:** within 48 hours  
- **Initial assessment:** within 3–5 business days  
- **Fix timeline:** depends on severity (critical issues are prioritized immediately)

You will receive updates as we investigate and resolve the issue.

---

### 🔒 What happens after reporting

#### If the vulnerability is accepted:
- We will confirm the issue privately
- Work on a fix immediately
- Release a patched version
- Optionally credit you in release notes (if you wish)

#### If the vulnerability is declined:
- We will explain why it is not considered a security risk
- Provide clarification or recommendations if needed

---

### ⚠️ Responsible Disclosure Guidelines

To ensure user safety and system integrity, we ask that you:

- Do NOT publicly disclose vulnerabilities before a fix is released  
- Avoid accessing or modifying user data beyond what is necessary for testing  
- Do NOT perform destructive or disruptive testing on live data or users  
- Report issues privately and responsibly
